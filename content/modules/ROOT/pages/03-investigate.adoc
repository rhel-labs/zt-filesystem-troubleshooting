= Investigating Disk Space

== Understanding the Storage Stack

Before we can fix the problem, we need to understand how storage is configured on this system. In RHEL, storage typically follows this hierarchy:

1. *Physical Disks* (or virtual disks in VMs)
2. *Physical Volumes (PVs)* - Disks initialized for use with LVM
3. *Volume Groups (VGs)* - Collections of physical volumes
4. *Logical Volumes (LVs)* - Flexible partitions created from volume groups
5. *Filesystems* - The formatted storage that applications use

== Step 1: View Block Devices

Let's start by looking at all the block devices (disks) on the system:

[source,bash,role=execute]
----
lsblk
----

This command shows all block devices in a tree structure. Look carefully at the output.

[TIP]
====
You should see a single 30GB disk containing:
- The operating system partitions
- LVM physical volumes
- Multiple logical volumes including your `/app` volume
====

Notice the LVM hierarchy - the disk contains physical volumes, which form volume groups, which contain logical volumes.

== Step 2: Examine Physical Volumes

Let's look at the physical volumes (PVs) in the LVM configuration:

[source,bash,role=execute]
----
pvs
----

Or for more detailed information:

[source,bash,role=execute]
----
pvdisplay
----

The `pvs` command shows a summary, while `pvdisplay` provides detailed information about each physical volume.

== Step 3: Check Volume Groups

Now let's examine the volume groups (VGs):

[source,bash,role=execute]
----
vgs
----

For more details:

[source,bash,role=execute]
----
vgdisplay
----

[IMPORTANT]
====
Look at the output carefully. You should see information about:
- *VG Size*: Total size of the volume group
- *PE Size*: Physical Extent size (the unit of allocation)
- *Total PE*: Total number of extents
- *Alloc PE*: How many extents are allocated
- *Free PE*: How many extents are still available

Pay special attention to the "Free PE" value - this shows unallocated space in the volume group!
====

== Step 4: Check Logical Volumes

Let's see the logical volumes:

[source,bash,role=execute]
----
lvs
----

Or with more detail:

[source,bash,role=execute]
----
lvdisplay
----

== The Discovery!

By now, you should have discovered something important:

✓ The `/app` logical volume is only 1GB in size

✓ The `/app` filesystem is completely full

✓ But the volume group has several GB of free space available!

✓ Scott only allocated 1GB to the app_lv when he could have used much more

This means Scott created a tiny logical volume to "save space" but there's plenty of unallocated space just sitting in the volume group, ready to be used!

[quote, Manager Scott (looking over your shoulder)]
____
"What do you mean 'free space in the VG'? I thought I used all the space efficiently! Are you saying I should have made the LV bigger? Well, obviously I knew that, I was just... testing the monitoring system. Yeah, that's it."
____

*Narrator*: Sure, Scott. You were definitely testing the monitoring system.

== The Solution Path

Now that you understand the problem, the solution is clear:

1. Extend the logical volume to use the free space in the volume group
2. Resize the filesystem to use the new LV space

That's it! No need to add disks or resize physical volumes - the space is already there in the VG, we just need to allocate it to the LV.
