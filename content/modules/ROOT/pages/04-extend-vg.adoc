= Verifying Available Space

== The Plan

Now that we've identified the problem (the LV is only 1GB but the VG has lots of free space), let's verify exactly how much space is available before we extend the logical volume.

== Step 1: Check Current Volume Group Status

Let's look at exactly how much free space is in the volume group:

[source,bash,role=execute]
----
vgs
----

Look at the "VFree" column - this shows how much unallocated space is available.

For more detailed information:

[source,bash,role=execute]
----
vgdisplay
----

[IMPORTANT]
====
Pay attention to these key fields:
- *VG Size*: Total size of the volume group
- *Alloc PE / Size*: How much is currently allocated to logical volumes
- *Free PE / Size*: How much is available for expansion

You should see several GB of free space ready to use!
====

== Step 2: Check Current Logical Volume Status

Now let's see the current size of our `/app` logical volume:

[source,bash,role=execute]
----
lvs
----

Look for the `app_lv` volume - it should be approximately 1GB.

For more details:

[source,bash,role=execute]
----
lvdisplay | grep -A 10 app_lv
----

== Step 3: Verify the Filesystem

Let's confirm the current filesystem size:

[source,bash,role=execute]
----
df -h /app
----

This should show approximately 1GB total size with nearly 100% used.

== Understanding the Situation

What we've confirmed:

✓ The volume group has plenty of free space (several GB)

✓ The logical volume for /app is only 1GB

✓ The filesystem is full (100% used)

✓ All we need to do is extend the LV to use the VG's free space!

[NOTE]
====
This is a common scenario in production environments. Storage is provisioned with extra capacity in the VG, but individual logical volumes start small and grow as needed. This is one of the key advantages of LVM - flexible, online storage growth!
====
